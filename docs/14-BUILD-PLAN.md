A granular, dependency-ordered implementation sequence. Where [10-ROADMAP.md](10-ROADMAP.md)
defines *product milestones and acceptance*, this doc defines the *order to write code* and
how each step is verified ‚Äî sized so **each task is one focused Claude Code session** ending

## ‚≠ê Re-point to the vision (current priority ‚Äî read [CLAUDE.md](../CLAUDE.md) "Product vision" first)

The app is built and runs, but the **voice is wired to the wrong default**: proactive call-outs and
many answers play **templated/canned** lines instead of being **LLM-generated from the live data**.
The north star (CLAUDE.md): _every utterance is generated by the AI from the real numbers and
the voice only speaks it; template is a **degraded fallback only**. There is no reflex/pre-rendered
tier ‚Äî nothing is pre-rendered._

**Acceptance test ‚Äî judge the whole product against this:** in a live session with a model running,
(a) hold PTT and ask ‚Üí the answer is **LLM-generated from tool data** and spoken; (b) the engineer
**proactively** keys the radio with **LLM-generated** lines (not canned) when something is genuinely
worth saying, gated so it isn't chatty. Nothing is pre-rendered.

**Do this re-point before more breadth ‚Äî packaging / UI reskin / extra strategy math all wait:**
1. **Make the LLM the default brain, not opt-in.** The free profile runs a **local model (Qwen via
   Ollama)** ‚Äî that *is* the AI; template mode is only for when no model resolves. (docs/15 free-first
   still holds: **local ‚âÝ template**.)
2. **Flip proactive phrasing to the LLM by default** in the voice layer ‚Äî **DONE (2026-06-16).** Rather
   than the payload-only `llmPhraser`, proactive call-outs now route through the **same tool-driven
   brain as the Q&A path**: `runProactiveTurn` (`packages/ai/src/proactive.ts`) hands the flagged event
   to the engineer, which reads the **whole** live situation via its read-only tools (session phase,
   lap, tyre/handling/fuel state, what just changed), reasons about the *cause*, and decides whether and
   what to say ‚Äî or returns the `SILENT` sentinel to stay quiet. `EngineerVoice` (`apps/desktop/src/
   voice-engine.ts`) defaults the proactive phraser to `engineerPhraser` (`packages/radio`) whenever a
   provider is configured; `templatePhraser` is the degraded fallback (no provider / no telemetry yet /
   provider error / cost cap). This is *stronger* than the original "payload + RaceState digest" plan ‚Äî
   the engineer pulls exactly the data it needs through the audited read-only tools, so "LLM never
   computes numbers" holds by construction (it reuses `runRadioTurn`).
3. **Proactivity = judgment, not rigid thresholds.** Keep the event rules as *candidate* triggers, but
   let the LLM (background strategist, docs/06) decide whether a candidate is worth voicing and generate
   the line; strong confidence-gating + quiet windows so it speaks **only when useful** (the driver's
   complaint was canned + spammy).
   - **The judgment now has a home: the _engineer skill_** (`packages/ai/src/skill.ts`, `ENGINEER_SKILL`,
     docs/06 ¬ßThe engineer skill) ‚Äî a per-domain reasoning playbook composed into every system prompt by
     `buildSystemPrompt`. It encodes *when to speak, when to stay silent, and the "the driver already
     knows ‚Äî don't repeat it" memory rule*. **Done:** the skill is written, wired into the prompt, and
     tested ‚Äî and now also composed into the **proactive** prompt (`buildProactiveSystemPrompt`), with a
     new doctrine line: cold tyres on the formation/out-lap/first-laps are *expected* ‚Äî reframe ("they'll
     be cold for the start, ease in ‚Äî I'll tell you when they're in") and follow up once they're in, don't
     alarm. **Wired (2026-06-16):** Tier-1+ candidate events route to the tool-driven LLM
     (`runProactiveTurn`), which pulls the live state itself via read-only tools instead of a static
     digest; `templatePhraser` / `apps/desktop/src/callout.ts` `SPOKEN_PHRASES` stay as the no-model
     fallback. The hot-path `onEvent` hook (`EngineerCore`) fires candidates off-throttle ‚Äî that's the
     trigger. A new edge-triggered rule `tireTempRecoveredRule` (`tire_temp_recovered`) gives the engineer
     the "tyres are up to temp now" follow-up moment for the start-of-stint arc.
4. **Validate on real data + a real model** (`pnpm dev:lmu` + a pulled Ollama model) ‚Äî never synthetic
   + template, which is what made it feel like a worse CrewChief.
5. **No reflex/pre-rendered tier** ‚Äî every utterance is LLM-generated; traffic awareness is
   anticipatory only (no instant "car alongside" proximity call).

Everything below (M0‚ÄìM11, packaging, polish) is the *substrate* the vision sits on. This section is the
lens it serves: **if a task doesn't move the acceptance test, it waits.**

## How to use this plan

- **One task per session.** Pick the lowest-numbered unblocked task. Load the docs listed
  in its **Context** line, implement, make its **Verify** step pass, commit.
- **Branch + commit.** This repo isn't a git repo yet ‚Äî task **T0.1** initializes it.
  Thereafter: one branch per task, conventional commits (`feat:`, `fix:`, `test:`‚Ä¶), small
  PRs.
- **Definition of Done (global):** `pnpm typecheck && pnpm lint && pnpm test` are green; new
  logic has unit tests; no secrets committed; docs updated if behavior diverges from them.
- **Replay-first verification.** Most tasks are verified offline against recorded or
  synthetic telemetry ‚Äî no game required. Tasks that *do* need LMU, a wheel, audio, or API
  keys are marked **[human-assisted]** and call out exactly what the user must do.
- **Free/local-first.** Build the free providers first (faster-whisper, Piper/Kokoro, local
  Qwen / free cloud tier / template mode); cloud Claude and other paid providers are opt-in,
  bring-your-own-key, added behind the same interface later. Never commit a key or add a
  central server. See [15-COST-AND-FREE-OPERATION.md](15-COST-AND-FREE-OPERATION.md).

## The central ordering idea

```
Native + game-dependent work (the spikes) needs a Windows PC + LMU + a wheel.
Everything else (schema, strategy, events, AI, voice logic, UI) is platform-independent
and testable against fixtures ‚Äî IF a replay/mock data source exists first.

  ‚áí Build the canonical schema + a synthetic/replay data source FIRST (M0).
    That unblocks ~70% of the codebase to be built and tested with no game running,
    in parallel with the human-assisted spikes (M1).
```

**Critical path:** `M0 scaffold+schema+replay ‚Üí M3 fuel/events ‚Üí M5 AI radio loop ‚Üí MVP`.
The LMU adapter (M1/M2) feeds real data in, but the logic above it is developed against
recordings, so a delay getting the plugin working does not block strategy/AI/UI work.

## Parallelization

Two tracks can run at once:
- **Track A (Claude Code, any OS):** M0 ‚Üí M3 ‚Üí M5 ‚Üí M6 logic against synthetic/recorded data.
- **Track B (human + Claude Code on the Windows rig):** M1 spikes ‚Üí M2 real adapter, then
  hand a real recording to Track A to replace synthetic fixtures.

They converge at the MVP gate (end of M5).

## M11 ‚Äî Virtual Energy (LMU binding constraint) ¬∑ _correctness fix, user-flagged_

The fuel/strategy engine originally modelled consumption in **litres only**. In LMU a stint
is frequently bound by the per-stint **Virtual Energy** budget, not fuel ‚Äî a car can have
litres left but be out of VE. Planning on litres alone is wrong for LMU. This makes VE a
first-class quantity parallel to fuel, so the binding stint/finish constraint is
`min(fuel-limited laps, VE-limited laps)`. Context: [05-STRATEGY-ENGINE](05-STRATEGY-ENGINE.md)
¬ß"Virtual Energy", [03-LMU-INTEGRATION](03-LMU-INTEGRATION.md) ¬ßVE / `/rest/strategy/usage`.
